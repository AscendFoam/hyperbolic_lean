# 第 1 个月执行清单

## 1. 本月目标

第 1 个月不追求模型结果，目标只有四个：

1. 跑通 LeanDojo 或等价的 Mathlib trace 获取链路
2. 明确 declaration graph 的最小可行 schema
3. 在一个小子域上完成一次抽图试运行
4. 拿到第一版 baseline 计划和实验记录模板

如果这四件事没完成，就不应该进入双曲模型阶段。

## 2. 本月交付物

月底之前至少要落地以下文件或结果：

1. 一份固定版本清单
2. 一份 declaration/edge/label 的 schema 文档
3. 一次真实的 trace 目录 inventory 结果
4. 一份小规模 declaration graph 样本
5. 一份 baseline 任务清单
6. 一份风险记录

## 3. 周计划

## 第 1 周：环境与边界锁定

### 目标

把“项目要依赖什么版本、做什么子图、第一批不做什么”全部定死。

### 必做事项

1. 锁定 Lean 4 版本
2. 锁定 Mathlib commit
3. 确认 LeanDojo 版本或替代 trace 方案
4. 明确第一个实验子域  
   建议优先：
   - `Mathlib/Algebra`
   - 或 `Mathlib/Topology`
5. 建立最小目录结构  
   至少包括：
   - `data/raw`
   - `data/processed`
   - `artifacts`
   - `logs`

### 本周产出

1. `versions.md` 或等价版本记录
2. 一份“首批子域选择说明”
3. 一个可运行的环境检查脚本或命令记录

### 验收条件

1. 后续所有实验都知道要基于哪个 commit
2. 不再出现“先全量跑一遍再说”的模糊策略

### 风险观察点

1. LeanDojo 环境安装失败
2. Mathlib 版本和 LeanDojo 版本不兼容
3. GPU/CPU/磁盘空间不足

### 若本周失败

先不要继续推进模型，优先做：

1. 用更小的 Mathlib 子集试跑
2. 调整到可获得 trace 的版本组合
3. 把第 2 周目标顺延，不进入 schema 细化

## 第 2 周：数据 schema 与 inventory

### 目标

先不急着抽全量图，而是先回答：

1. trace 目录里到底有什么文件
2. 我们要抽成什么字段
3. 哪些字段是必须的，哪些是后补的

### 必做事项

1. 跑一遍 trace 目录 inventory
2. 统计：
   - 文件类型
   - 文件数量
   - 目录层级
   - 样本文件大小
3. 对照 schema 文档，确认最小字段集
4. 定义 ID 规则  
   例如：
   - `declaration_id`
   - `edge_id`
   - `module_id`

### 本周产出

1. inventory 结果
2. 第一版 schema 文档
3. 一份字段映射草表  
   例如：
   - trace 字段 -> declaration 字段
   - trace 字段 -> edge 字段

### 验收条件

1. 至少知道 declaration graph 需要的原始数据能否从 trace 中拿到
2. schema 已经可以指导脚本输出 CSV/Parquet

### 风险观察点

1. LeanDojo 导出的格式和预期不一致
2. declaration 与 proof-state 信息分散在不同文件中
3. 没法稳定拿到 declaration 依赖

### 若本周失败

可以临时把目标降级为：

1. 先做 trace 目录清点
2. 手工解析少量样本文件
3. 先只设计 declaration-level schema，不碰 proof-state schema

## 第 3 周：小规模 declaration graph 抽取

### 目标

在一个小子域上拿到第一版真实 graph 数据，而不是继续停留在纸面设计。

### 必做事项

1. 选一个小子域  
   建议不超过 1000-5000 个 declaration
2. 运行抽图脚本
3. 输出至少三张表：
   - declarations
   - edges
   - manifest/stats
4. 做基本质量检查：
   - 重复 ID 数量
   - 空字段比例
   - 边的 src/dst 覆盖率
   - 孤立点比例

### 本周产出

1. 第一版 graph 样本数据
2. 数据质量报告
3. 一份失败样本清单

### 验收条件

1. 至少有一版图数据可以被下游 baseline 读取
2. 数据质量问题已知，而不是“先留着以后再看”

### 风险观察点

1. declaration 粒度不一致
2. 某些边类型混淆
3. 引用关系与继承关系无法区分

### 若本周失败

不要盲目扩大规模，先做：

1. 只保留一种边类型
2. 只保留一种节点族
3. 先得到极简图，再逐步加字段

## 第 4 周：baseline 设计与 Gate A 评估

### 目标

月底时应该回答：

1. 数据是否可用
2. 下一步是否值得进入 baseline 建模

### 必做事项

1. 明确首批 baseline 任务：
   - graph structure diagnostics
   - link prediction
   - declaration type classification
2. 明确首批 baseline 模型：
   - heuristic
   - Node2Vec/DeepWalk
   - GCN/GraphSAGE
3. 用模板建立实验记录文件
4. 做第 1 次 Gate A 评审

### 本周产出

1. baseline 任务清单
2. baseline 结果记录表
3. Gate A 评估结论

### Gate A 判据

以下 4 条至少满足 3 条，才建议进入第 2 个月：

1. 版本组合固定且可复现
2. 小子域 graph 可稳定抽出
3. declaration 和 edge schema 已经定型
4. 至少一种标签族具备后续可行性

### 若 Gate A 未通过

第 2 个月不要进入双曲模型，先补：

1. 数据抽取稳定性
2. schema 修订
3. 标签策略收缩

## 4. 每周固定动作

每周都建议固定做下面 5 件事：

1. 更新一次实验日志
2. 更新一次风险清单
3. 保存一次数据快照
4. 记录一次失败案例
5. 形成一次导师讨论材料

## 5. 每周日志模板

建议每周日志固定包含：

1. 本周目标
2. 本周完成
3. 本周未完成
4. 当前 blocker
5. 下周计划
6. 是否影响 stage gate

## 6. 本月结束时必须回答的 8 个问题

1. 我们到底能否稳定拿到 declaration graph？
2. 第一批图到底选哪个子域？
3. 哪些边类型是可信的？
4. 哪些标签是高置信的？
5. proof-state 图是否要立刻做，还是延后？
6. 第一个 baseline 任务是什么？
7. 第一个欧氏 baseline 模型是什么？
8. 第 2 个月是进入建模，还是继续补数据？

## 7. 建议的月底汇报结构

月底汇报建议只讲四件事：

1. 数据拿到了什么
2. 数据哪里不稳
3. 下个月具体做哪三个 baseline
4. 当前最可能失败的点是什么

只要这四件事讲清楚，这个月就是成功的。
