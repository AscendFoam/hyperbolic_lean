# 形式化证明工程化实验方案（修订版）

## 0. 修订说明

本修订版是在阅读 [Gemini的调研报告](d:\Codes\Math\hyperbolic_lean\docs\Gemini的调研报告.md)、[Doubao的调研报告](d:\Codes\Math\hyperbolic_lean\docs\Doubao的调研报告.md) 以及已有的 [形式化证明工程化初步实验方案](d:\Codes\Math\hyperbolic_lean\docs\形式化证明工程化初步实验方案.md) 后重写的版本。

与初版相比，这一版有六个关键更新：

1. **收紧主线问题**  
   初版同时推进“表达式级显式编码、形式化图谱、证明轨迹”三条主线，主题过宽。修订版明确把可投稿主线收束为：  
   **形式化数学图谱的双曲表示 + 性质 probing + 检索辅助证明**。

2. **把表达式级实验降级为支撑模块**  
   表达式重写审计器仍然保留，但定位从“并列主线”改为“低成本验证显式维度思想的前期试验台”。

3. **把 `γ` 无理性探索降级为远期探索，不再作为主项目目标**  
   两份调研都给出了很多启发，但从工程与投稿可行性看，`γ` 相关内容最多只能作为后期案例分析或附录式探索，不能作为主论文承诺。

4. **吸收了更具体的数据工程思路**  
   尤其是 LeanDojo、Graph2Tac、LeanProgress、Lean 元编程/类型类综合等方向带来的启发。  
   其中最值得吸收的是：**用 Lean 自己做标签抽取与验证，而不是靠文本规则硬猜。**

5. **增加阶段门（stage gate）和备选路径**  
   初版偏“路线图”；修订版加入了 go/no-go 判据、失败分支和降级投稿路线，避免项目在中途失控。

6. **加强严谨性约束**  
   修订版明确：
   - 不把 `synthInstance` 失败直接当作负标签
   - 不默认整个 Mathlib 都适合双曲建模
   - 不一开始就上最复杂的 HypHGT/长证明生成
   - 不把“几何解释”当作先验结论，而是当作待验证假设

## 1. 立项结论

### 1.1 最终建议

如果目标是做出**一项真实可执行、可复现、可投稿**的成果，那么最合理的项目定位应当是：

**构建一个面向 Lean/Mathlib 的形式化数学图谱表示学习平台，验证双曲表示是否比欧氏表示更适合建模其层级结构，并进一步评估这种表示是否有助于性质 probing 和 proof-hint retrieval。**

这条主线的优点是：

1. 有明确数据源  
   LeanDojo 和 Mathlib 已经提供了较干净的形式化语料基础。

2. 有自然任务闭环  
   图抽取、嵌入学习、性质预测、检索、Lean 验证可以构成可重复实验链路。

3. 有足够的新意但不过分冒进  
   “形式化数学图谱 + 双曲表示 + 性质 probing/检索”这个交叉点仍然比较稀缺，但比“自动发现未解难题证明”现实得多。

4. 便于形成论文叙事  
   论文可以围绕“数学形式化图谱具有层级友好的几何结构”展开，而不是被多个方向稀释。

### 1.2 不建议的主目标

以下方向在当前阶段不应作为主立项目标：

1. 自动证明 `γ` 无理性或其他开放问题
2. 直接构造端到端长证明生成系统
3. 先做全网自然语言自动形式化，再做双曲图谱
4. 同时把表达式显式编码、图谱嵌入、证明搜索、自然语言对齐都做成同等主线

## 2. 核心研究问题与非目标

### 2.1 核心研究问题

修订后建议聚焦四个研究问题。

### RQ1：Mathlib 声明图是否具有足够强的层级性，从而值得使用双曲表示？

这里不能先假定“数学知识一定适合双曲空间”，而是应先做结构诊断：

- DAG 深度统计
- ancestor/descendant 分布
- tree-likeness
- Gromov hyperbolicity 或近似层级性指标
- 不同子领域子图的结构差异

如果这一层证据不足，后续双曲建模的说服力会很弱。

### RQ2：在低维表示下，双曲模型是否优于欧氏模型？

重点不只是看最终准确率，还要看：

- 低维压缩下的性能保持
- 层级边的恢复能力
- 检索质量
- 结构失真

换言之，要证明的是：

**双曲空间不是“也能做”，而是在特定任务上更合适。**

### RQ3：形式化数学对象的某些性质能否在嵌入空间中被可靠 probe 出来？

例如：

- declaration type: `def/theorem/lemma/class/instance`
- 某些代数/拓扑类型类属性
- 某声明所属的数学子域或模块簇

这里要把“显式特征维度”从哲学直觉变成可测实验。

### RQ4：图谱表示是否能帮助 proof-hint retrieval，而不仅仅是做可视化？

如果最终只能画出漂亮的图，而不能提升任何与证明有关的检索或排序任务，论文价值会明显下降。

因此至少要验证以下一项：

- theorem retrieval
- proof-state nearest-neighbor retrieval
- tactic ranking
- compile-success 提升

### 2.2 非目标

本项目当前不试图证明：

1. 双曲空间一定能学出“真实的数学语义轴”
2. 嵌入空间的局部几何簇一定对应深层数学真理
3. 模型能自动发现开放问题证明
4. 表达式显式编码本身就足以支撑完整形式化证明

这些都只能作为探索性结论，而不是主论文 claim。

## 3. 修订后的总架构

修订版建议采用“一条主线 + 两个支撑模块”的结构。

### 3.1 主线模块 M：形式化数学图谱的双曲表示学习

这是主论文方向，包含：

1. Mathlib/LeanDojo 图数据抽取
2. 图结构诊断
3. 欧氏 baseline
4. 双曲 baseline
5. 性质 probing
6. retrieval 评估

### 3.2 支撑模块 S1：proof-state 检索与 tactic 轻量排序

这是与“证明”最近的一层，但仍然保持在检索/排序，而不是生成。

其作用是：

1. 让主线与自动定理证明社区更接轨
2. 为投稿提供更强的应用价值
3. 在不做长证明生成的前提下，验证表示是否真正有用

### 3.3 支撑模块 S2：表达式级多视角显式编码试验台

这一模块不作为主论文中心，而作为：

1. 显式特征思想的低成本验证器
2. 未来做 proof-state 多视角编码的原型来源
3. 若主线遇阻时的备用小成果产出点

### 3.4 远期探索模块 X：`γ` 等特殊常数案例分析

定位为：

1. 展示系统如何用于“数学研究启发”
2. 作为 future work 或附录案例

不定位为：

1. 论文主结果
2. 项目交付承诺

## 4. 数据工程方案

## 4.1 数据源与版本锁定

所有实验必须固定在单一、可复现的版本组合上：

- Lean 4 版本
- Mathlib commit
- LeanDojo 版本
- Python 环境与模型依赖版本

建议在项目启动时就生成：

1. `environment.yml` 或 `requirements.txt`
2. 数据抽取脚本版本号
3. 数据卡（data card）
4. 仓库 commit 记录表

否则后面图结构和标签会随着 Mathlib 更新而漂移，导致实验不可复现。

## 4.2 图对象的选择

初版不建议“一张大图包打一切”，而建议拆成三类图，逐层推进。

### 图 A：声明依赖图

节点：

- `def`
- `theorem`
- `lemma`
- `class`
- `instance`

边：

- declaration 使用 declaration
- theorem proof 中引用 declaration
- definition body 中引用 declaration

这是主线图。

### 图 B：类型类继承图

节点：

- typeclass
- instance
- 结构体

边：

- extends
- instance-of
- coercion-related relation

这是最有可能天然表现出层级结构的子图，应优先验证双曲优势。

### 图 C：proof-state 转移图

节点：

- proof state

边：

- tactic transition

这是后续 retrieval/tactic 任务的图基础，不必在最早阶段全量抽取。

## 4.3 标签抽取：吸收调研报告后的修正版

两份调研里最值得吸收的工程点，是**利用 Lean 元编程和类型类综合来提取标签**。  
但初版里有一个需要严格修正的地方：

**`synthInstance` 失败不能直接记为负标签。**

因为失败可能意味着：

1. 查询类型本身不适用于该声明
2. 实例搜索超时
3. 属性不可由 typeclass 直接表达
4. 当前环境缺少桥接实例

因此，标签必须采用**三值制**：

- `positive`
- `negative`
- `unknown`

### 4.3.1 推荐的标签来源

1. **结构性标签**
   - declaration kind
   - namespace/module
   - file path
   - whether theorem/definition/class

2. **类型类可综合标签**
   - `CommGroup`
   - `CommRing`
   - `Finite`
   - `CompactSpace`
   - 其他仅在查询类型正确且实例语义稳定时使用

3. **语法/复杂度标签**
   - AST 大小
   - proof 长度
   - dependency depth

### 4.3.2 推荐的标签策略

对每个候选属性 `P`：

1. 先判断该声明是否能形成合法查询
2. 若能合法查询且 `synthInstance` 成功，则标 `positive`
3. 若能合法查询且存在显式反例或显式不可满足证据，才标 `negative`
4. 其余全部标 `unknown`

对于 `negative` 难以稳定构造的属性，建议只做：

- `positive` vs `unlabeled`
- 或只在高置信子集上做二分类

这会牺牲一部分数据量，但会显著提高结论可信度。

## 4.4 数据切分策略

为避免泄漏，不能随机切声明。

建议优先尝试三种切分：

1. **模块切分**
   - 按 `Mathlib/Algebra/...`、`Mathlib/Topology/...` 等目录切

2. **时间切分**
   - 若能拿到不同版本轨迹，可按 commit 时间切

3. **边切分**
   - link prediction 时做边遮蔽，但必须保持训练图连通性

若只做随机切分，很容易学到文件级模式而非真正的结构泛化。

## 4.5 首批数据规模建议

不要一上来全量跑整个 Mathlib。

建议按以下顺序推进：

1. 类型类继承相关子图
2. `Algebra` + `Topology` 两个子域
3. 再扩展到更大全量声明图

这是因为：

1. 这些子图层级性更明显
2. 标签更容易抽
3. 更适合作为双曲表示的首批验证场景

## 5. 模型路线：先稳，再难

## 5.1 不要一开始就上最复杂模型

Gemini 报告中的 HypHGT/HypTransformer 类方向有吸引力，但对当前项目而言，**过早使用复杂模型只会增加调参难度和审稿质疑点**。

更合理的模型梯度是：

### 第一级：图启发式与浅层基线

- degree / PageRank / ancestor overlap
- BM25 / lexical retrieval
- Node2Vec / DeepWalk

作用：

1. 给出最小可比基线
2. 验证任务本身不是“随便做都很高分”

### 第二级：欧氏图模型

- GCN
- GraphSAGE
- GAT
- 简单的 graph transformer

作用：

1. 建立强欧氏 baseline
2. 为后续双曲模型提供公平对照

### 第三级：双曲模型

- Poincaré embedding
- HGCN
- Lorentz / hyperbolic GNN 变体

### 第四级：可选的高复杂模型

- HypHGT / Hypformer / 更复杂双曲 transformer

只有在前三级已经形成清晰证据时，才值得上这一级。

## 5.2 为什么这样分层

因为论文要回答的不只是“能不能做”，而是：

1. 简单欧氏方法做到哪一步
2. 双曲方法在哪个点开始带来优势
3. 这个优势是来自模型容量，还是来自几何归纳偏置

如果没有清晰的 baseline 梯度，审稿人很容易把所有收益归因于“模型更复杂”。

## 6. 实验任务设计

## 6.1 T0：图结构诊断

这是修订版新增的必做任务。

目标：

在投入双曲模型之前，先证明图结构至少部分具有双曲友好性。

指标：

- graph depth
- branching factor
- shortest-path statistics
- tree-distance distortion
- approximate Gromov hyperbolicity
- ancestor/descendant recovery difficulty

预期结论：

1. 不是整个 Mathlib 都一样层级化
2. 某些子图比另一些更适合双曲表示

这会直接决定后续实验该在哪些子图上做。

## 6.2 T1：声明依赖边的 link prediction

输入：

- 部分遮蔽后的声明依赖图

输出：

- 被遮蔽依赖边的预测排序

指标：

- MRR
- Hits@1 / Hits@10
- MAP
- AUROC

这是最标准、最容易形成主论文结果的任务。

## 6.3 T2：节点类型与性质 probing

任务分两类：

### 稳定标签任务

- `def/theorem/lemma/class/instance`
- module/domain 分类
- file-level family 分类

### 高价值标签任务

- `CommGroup` 类属性
- `CommRing`
- `Finite`
- `CompactSpace`

指标：

- macro F1
- AUROC
- calibration
- probe complexity sensitivity

这里建议加入：

1. 线性 probe
2. MLP probe

如果线性 probe 已经有效，说明表示空间更有解释力。

## 6.4 T3：检索任务

建议至少做两个：

### theorem retrieval

给定一个 theorem 或 goal 描述，检索相关声明。

### proof-state retrieval

给定当前 state，检索历史上相似 state 和成功 tactic。

基线：

- BM25
- dense text retrieval
- graph embedding retrieval
- graph + text hybrid retrieval

指标：

- Recall@k
- MRR
- nDCG
- top-k useful-hit rate

## 6.5 T4：轻量 tactic ranking

这不是长证明生成，而是：

给定 proof state 和候选 tactics，排序哪个更可能成功。

可以借鉴：

- Graph2Tac 的表示思路
- LeanProgress 这类 proof-progress 指标思路

指标：

- top-k accuracy
- MRR
- compile-success@k

如果这个任务能带来稳定提升，你的论文应用价值会明显增强。

## 6.6 T5：表达式级显式编码试验台

此任务不作为主论文核心，但建议保留。

作用：

1. 快速验证“多视角显式编码”是否能学到
2. 为后续 proof-state 多头表示提供先验

建议任务：

- `truth_preserved`
- `domain_preserved`
- `operator_changed`
- `structure_changed`
- `explicitness_delta`

这部分最好作为：

- 项目先导实验
- 或附录补充实验

## 7. 评价标准与阶段门

修订版最重要的增强之一，是加入明确阶段门。

## Gate A：数据可行性门

判据：

1. 成功锁定 Lean/Mathlib/LeanDojo 版本
2. 在一个子域上稳定抽取声明图
3. 生成不少于 `10k` 声明节点的结构化数据
4. 至少一种标签族能稳定抽取

若失败：

1. 缩到更小子图
2. 先只做声明类型分类与 link prediction
3. 暂停性质 probing

## Gate B：欧氏基线门

判据：

1. 欧氏图模型显著优于随机/浅层启发式
2. 至少一个任务形成稳定、可复现实验表格

若失败：

1. 说明任务定义或数据质量有问题
2. 先不进入双曲阶段

## Gate C：双曲价值门

判据：

以下至少满足其一：

1. 双曲模型在低维下优于欧氏模型
2. 双曲模型在相同精度下需要更少维度
3. 双曲模型在层级相关任务上显著更优

若失败：

1. 降级为“形式化数学图谱 benchmark / dataset / probing”论文
2. 将双曲部分降为附加实验，而非中心 claim

## Gate D：应用价值门

判据：

1. 检索或 tactic ranking 至少有一个子任务优于文本基线
2. 或至少能证明图表示提供互补信息

若失败：

主论文仍可投稿，但故事线应转为：

**formal math graph representation benchmark**

而不是 proof assistance。

## 8. 从立项到投稿的执行计划

这里给出一条更稳妥的 `36 周主路径 + 8 周缓冲` 计划。

## 阶段 0：立项与复现实验准备（第 1-2 周）

目标：

1. 锁定版本
2. 明确研究 claim
3. 明确资源边界

交付物：

- 项目章程
- 技术栈清单
- 资源预算
- 版本锁定文件

## 阶段 1：数据抽取试运行（第 3-6 周）

目标：

1. 跑通 LeanDojo
2. 先在小子域上抽图
3. 验证能否拿到 declaration-level 结构

交付物：

- 小规模图样本
- 抽取脚本
- 数据字段说明

阶段门：

- Gate A

## 阶段 2：数据集正式构建（第 7-10 周）

目标：

1. 构造主图
2. 构造可靠标签
3. 做初步统计分析

交付物：

- 主图数据集 v1
- 标签数据集 v1
- 图结构统计报告

## 阶段 3：欧氏 baseline（第 11-14 周）

目标：

1. 先把任务做通
2. 建立强 baseline

交付物：

- heuristic baseline
- Node2Vec/GCN/GraphSAGE 结果
- baseline 表格 v1

阶段门：

- Gate B

## 阶段 4：双曲 baseline（第 15-20 周）

目标：

1. 跑通 Poincaré/HGCN
2. 比较维度-性能曲线
3. 确认双曲是否真的有价值

交付物：

- 双曲实验表格
- 维度压缩对比图
- 双曲 vs 欧氏误差分析

阶段门：

- Gate C

## 阶段 5：probes 与 retrieval（第 21-26 周）

目标：

1. 做稳定的 node/property probing
2. 做 theorem/proof-state retrieval
3. 若可行，再加 tactic ranking

交付物：

- probing 结果
- retrieval 结果
- demo 原型

阶段门：

- Gate D

## 阶段 6：消融、稳健性与失败案例（第 27-30 周）

目标：

1. 做 ablation
2. 做不同子图对比
3. 整理失败案例

必须做的消融：

1. 图类型消融
2. 标签族消融
3. 维度消融
4. 文本特征与图特征的融合消融
5. 双曲曲率/模型复杂度消融

交付物：

- ablation 表
- 误差分析报告
- 失败案例附录

## 阶段 7：论文写作与投稿包准备（第 31-36 周）

目标：

1. 写论文
2. 整理开源材料
3. 形成投稿包

投稿包应包含：

1. 主论文
2. appendix
3. artifact README
4. data card
5. model card
6. 可复现脚本
7. 失败案例说明

## 缓冲阶段：返修与备选投稿（额外 8 周）

用于：

1. 根据导师反馈重写
2. 若主会不合适，转投滚动期刊或 workshop
3. 补实验

## 9. 投稿策略

## 9.1 主投稿故事线

推荐主标题方向：

**Hyperbolic Representations for Formal Mathematical Dependency Graphs**

或

**Geometry-Aware Representation Learning on Lean/Mathlib Formal Graphs**

核心 claim 应保持克制：

1. 我们构建了一个可复现的 formal math graph 数据管线
2. 我们证明某些 formal graph 子结构具有明显层级友好性
3. 双曲表示在低维下更适合这些子结构
4. 这种表示对 probing / retrieval 有实际价值

## 9.2 目标 venue 梯度

不写具体日期，只给类型梯度：

### A 档

- ICLR / NeurIPS / ICML 主会  
  条件：双曲结果清晰，retrieval 有稳定增益，实验完整

### B 档

- TMLR  
  条件：结果扎实、工程完整、artifact 强，但 novelty 不足以冲主会

### C 档

- Math-AI / AI for Science 相关 workshop  
  条件：数据集和初步结果成型，但主线结果尚不够强

## 9.3 最终投稿前的最低清单

投稿前至少要满足：

1. 一个公开可复现的数据抽取脚本
2. 一个强欧氏 baseline
3. 一个稳定双曲 baseline
4. 一组明确的 ablation
5. 至少一个应用任务结果
6. 对失败案例的诚实讨论

## 10. 风险分析与解决方案

## 10.1 数据风险

### 风险 1：Lean/Mathlib 版本变动导致图不稳定

解决：

1. 固定 commit
2. 将抽取结果版本化
3. 论文中明确数据快照版本

### 风险 2：标签抽取语义不稳

解决：

1. 采用三值标签
2. 只在高置信标签上做主实验
3. 明确 unknown 比例

### 风险 3：不同声明类型混在一起导致任务失真

解决：

1. 分图建模
2. 分类型评估
3. 在报告中区分 declaration family

## 10.2 模型风险

### 风险 4：双曲模型训练不稳定

解决：

1. 先从 Poincaré embedding 开始
2. 固定曲率后再尝试可学习曲率
3. 严格监控梯度、范数和数值溢出

### 风险 5：双曲并不优于欧氏

解决：

1. 先做图结构诊断，不盲目押注
2. 聚焦最层级化的子图
3. 若仍无优势，转成 benchmark/probing 论文

### 风险 6：复杂模型收益只是参数量带来的

解决：

1. 控制参数量
2. 比较等参数规模模型
3. 做维度-性能曲线而不只看单点最好结果

## 10.3 论文风险

### 风险 7：故事线太散

解决：

1. 主文只保留一条核心线
2. 表达式模块与 `γ` 案例放附录或 future work

### 风险 8：审稿人质疑“为什么一定要双曲”

解决：

1. 在正文前部加入图结构诊断
2. 用低维压缩与层级任务收益回答
3. 提供失败子图，说明我们没有过度泛化

## 10.4 工程风险

### 风险 9：LeanDojo 全量抽取过慢

解决：

1. 从子图开始
2. 先构建脚本缓存
3. 将 proof-state 和 declaration 图分开处理

### 风险 10：GPU 资源不足

解决：

1. 先做小图/低维实验
2. 优先浅层模型
3. 把大模型实验留到最后阶段

## 11. 资源与协作假设

如果该项目由一名研究生主做，比较合理的资源假设是：

1. 一台可用 GPU 服务器
2. Lean/Mathlib 环境长期稳定可访问
3. 至少每两周一次的导师讨论
4. 一名外部熟悉 Lean 或图学习的同学可偶尔交流

若上述条件明显不足，则项目应进一步收缩为：

**formal graph dataset + 欧氏/双曲 baseline + probing**

不要强行把 retrieval 或 tactic ranking 也一起做。

## 12. 对旧版方案的最终修正结论

综合两份调研报告后，我认为旧版方案最需要修正的地方有四点：

1. **主线要更窄**  
   旧版把三个方向并列推进，风险太大。  
   修订后：主线只保留 formal graph + hyperbolic + probing/retrieval。

2. **标签要更谨慎**  
   旧版强调“可做性质标签”，但没把 negative/unknown 区分讲透。  
   修订后：明确三值标签和高置信子集策略。

3. **模型梯度要更稳**  
   旧版容易让人直接跳到复杂双曲 transformer。  
   修订后：先浅层，再欧氏，再双曲，再高复杂模型。

4. **投稿路径要前置设计**  
   旧版讲了实验，但对主张强度与投稿梯度约束不够。  
   修订后：加入 stage gate、备选故事线和 venue 梯度。

## 13. 当前最优先的四个动作

如果从今天开始推进，我建议严格按下面顺序做：

1. 跑通 LeanDojo，在一个小子域上拿到 declaration graph
2. 定义节点/边 schema，并确认哪些属性能稳定抽标签
3. 先做图结构诊断和欧氏 baseline
4. 只有在 Gate B 通过后，再进入双曲模型阶段

这四步走完之前，不建议继续扩展到：

- 长证明生成
- 复杂双曲 transformer
- `γ` 无理性探索
- 大规模自然语言对齐

## 14. 总结

修订后的结论可以概括为一句话：

**这个项目最可行的工程化落地方式，不是“让 AI 直接证明数学难题”，而是先把 Lean/Mathlib 中的形式化数学对象变成一个可诊断、可嵌入、可检索、可验证的几何图系统。**

只要主线控制在：

1. formal graph 数据管线
2. 欧氏/双曲表示对比
3. probing 与 retrieval 的应用验证

那么它就是一个严谨、收敛、且真正有投稿希望的项目。

而表达式级显式编码与 `γ` 案例，则更适合作为：

- 先导试验
- 补充模块
- 后续扩展

而不是当前阶段的主战场。
