# 形式化证明工程化初步实验方案

## 1. 文档目的

本文档基于 [我和Gemini的对话](d:\Codes\Math\hyperbolic_lean\docs\我和Gemini的对话.md) 中提出的核心设想，尝试回答一个更工程化的问题：

**如果要把“多视角显式编码 + 形式化验证 + 双曲几何表示”这一思想做成一个可实现、可迭代、可发表的完整项目，第一版应该怎样设计？**

本文档的目标不是直接提出一个“自动证明黎曼猜想”式的大系统，而是把原始想法拆成几个可以独立验证的实验子问题，形成一个一年内有机会做出阶段性成果的项目方案。

## 2. 从对话中提取的关键信息

### 2.1 核心思想

对话中最重要的不是某个具体模型，而是下面这几个判断：

1. 数学表达式不能只被看成 token 序列，而应被看成具有多种信息视角的对象。
2. 这些视角至少包括：
   - 真值/解空间信息
   - 结构信息
   - 运算信息
   - 复杂度信息
   - 显式程度信息
   - 生成方式与收敛方式等过程信息
3. 数学推理不应只做“对/错”的二值判断，而应分析每一步变换在不同信息维度上改变了什么。
4. 形式化证明系统的价值在于：它可以提供可靠的“真值约束”和机器可验证的推理轨迹。
5. 数学知识具有明显层级结构，因此用双曲空间建模可能比欧氏空间更自然。

### 2.2 可以转化为工程目标的关键命题

从工程角度看，对话中真正能落地的命题主要有四个：

1. **表达式级不变量检测**  
   对于 `x + y = 1 -> y = 1 - x` 这样的变换，系统应判断：
   - 真值是否保持
   - 结构是否改变
   - 运算图是否改变
   - 显式程度是否提升

2. **形式化图谱的几何表示**  
   Lean/Mathlib 中的定义、定理、类型类、证明依赖关系可以抽取为图；该图具有强层级性，适合尝试双曲嵌入。

3. **证明轨迹的向量化建模**  
   在 Lean 中，一个 proof state 加上一条 tactic 可以看成一次状态变换。该变换可以被分析为“保持哪些维度不变、改变哪些维度”。

4. **AI 辅助证明不应直接端到端生成最终证明，而应先做 proof-hint retrieval 与中间状态分析**  
   这是把高风险目标降级为低风险、可验证目标的关键。

### 2.3 对原始设想的工程化修正

原始设想非常宏大，但如果直接做，会同时踩中四个高风险点：

1. 真值语义本身难以直接学习。
2. 自动形式化数据极易脏。
3. 双曲空间训练与检索工程复杂。
4. 端到端自动证明评价成本极高。

因此，工程上必须做三件事：

1. **先限制问题语言**，从表达式重写和 Lean 现成语料出发。
2. **先做混合表示**，不要一开始就完全依赖神经网络自动“悟出”全部特征。
3. **先做验证型系统**，再做生成型系统。

## 3. 项目定位

### 3.1 项目代号

可将项目暂定为：`Hyperbolic Lean`

### 3.2 项目总目标

构建一个面向数学表达式和形式化证明的实验平台，使系统能够：

1. 对表达式变换进行多维度变化分析。
2. 对形式化数学知识图谱进行欧氏/双曲对比建模。
3. 对 proof state 与 tactic 轨迹进行表示学习。
4. 在形式化验证闭环中，支持 proof hint 检索与路径分析。

### 3.3 第一阶段的真实目标

第一阶段不追求“自动发现全新证明”，而追求下面这个更现实的目标：

**做出一个能够在局部表达式变换、形式化知识图谱建模、证明轨迹检索这三个层面都给出可验证结果的原型系统。**

如果这一点做成，就已经具备论文雏形和后续扩展价值。

## 4. 核心研究问题重述

为了把概念落地，需要把问题改写成可监督或可评价的任务。

### RQ1：能否显式判断一步表达式变换改变了哪些信息维度？

输入一对表达式 `(E1, E2)`，输出：

- `truth_preserved`
- `domain_preserved`
- `operator_changed`
- `structure_changed`
- `complexity_delta`
- `explicitness_delta`

这对应你最原始的设想，也是最适合先验证的任务。

### RQ2：Mathlib 的形式化依赖结构是否天然更适合双曲表示？

输入 Lean/Mathlib 图谱，比较欧氏 GNN 与双曲 GNN 在以下任务上的表现：

- link prediction
- node classification
- hierarchical retrieval
- low-dimensional distortion

如果双曲空间在低维下显著更优，就说明“层级数学结构适合负曲率空间”这一假设成立。

### RQ3：proof state 与 tactic 能否被表示为“状态点 + 变换向量”？

输入：

- 当前 proof state
- 可用 tactic
- 下一 proof state

输出：

- tactic 类型预测
- tactic 是否为“等价重写型”还是“弱化/投影型”
- 检索到的近邻证明片段

这一步会把你的“推理是流形上的运动”转成可实验的问题。

### RQ4：是否能在形式化验证闭环中构建 proof-hint 系统？

输入一个 Lean goal，系统检索：

- 相似 theorem
- 相似 proof state
- 候选 tactic

然后由 Lean 编译器验证结果是否合法。

## 5. 总体技术路线

建议采用四层架构，而不是一个巨大的端到端模型。

### 5.1 数据层

包含两类数据源：

1. **表达式级数据**
   - 使用 SymPy 生成等价/不等价表达式对
   - 使用手写规则生成可控重写轨迹
   - 对每一步打上多维标签

2. **形式化证明级数据**
   - 使用 LeanDojo 抽取 Mathlib 中的定义、定理、依赖边、proof state、tactic trace
   - 仅使用已验证的形式化语料，不在第一阶段引入自动形式化

### 5.2 表示层

采用“显式特征 + 学习表示”的混合方案：

1. **显式特征**
   - AST 节点数、树高、变量数
   - 运算符统计
   - 是否解出了指定变量
   - 近似数值采样一致性
   - 域约束变化

2. **学习表示**
   - AST encoder
   - graph encoder
   - hyperbolic embedding
   - proof-state encoder

### 5.3 推理与检索层

1. 表达式对比器：输出多维变化报告
2. 图谱检索器：按欧氏或双曲距离召回近邻节点
3. tactic 候选器：按 proof state 相似性召回历史成功策略

### 5.4 验证层

这是整个系统与普通“数学 LLM”最大的区别：

1. 表达式级验证由 SymPy、数值采样和约束检查承担
2. 形式化级验证由 Lean 编译器承担

原则上，系统的每一个高层输出都应该能被下层验证模块复核。

## 6. 项目分解：三个实验子系统

我建议不要一开始就做“完整系统”，而是做三个逐步耦合的子系统。

### 子系统 A：表达式级多视角编码与重写审计器

#### 目标

验证“多维显式编码”是否真的能检测局部推理问题。

#### 输入

- 单个表达式
- 表达式对
- 表达式变换轨迹

#### 输出

- 语义是否保持
- 哪些维度发生改变
- 显式程度是否增加
- 是否存在可疑变换

#### 推荐的第一批规则

1. 交换律、结合律
2. 分配律
3. 移项、同类项合并
4. 展开与因式分解
5. 分式约分
6. 带域风险的变换  
   例如：
   - `sqrt(x^2) -> x`
   - `(x^2 - 1)/(x - 1) -> x + 1`

这类规则特别重要，因为它们能逼系统区分“表面等价”与“带条件等价”。

### 子系统 B：Mathlib 形式化图谱的双曲表示

#### 目标

验证双曲空间是否更适合承载形式化数学层级结构。

#### 节点候选

- definition
- theorem
- lemma
- class
- instance
- namespace/module

#### 边候选

- import
- dependency
- proof-use
- typeclass inheritance

#### 关键任务

1. link prediction
2. node retrieval
3. property probing
4. embedding visualization

### 子系统 C：proof state 与 tactic 的轨迹分析器

#### 目标

验证“证明状态是点，tactic 是变换向量”的假设是否有用。

#### 输入

- 当前 goal
- 本地上下文
- 已执行 tactic
- 下一状态

#### 输出

- tactic 类型
- 近邻历史状态
- 成功 proof fragment
- 变换性质标签  
  例如：
  - rewrite-like
  - simplification-like
  - weakening-like
  - decomposition-like

## 8. 初步实验设计

### 实验 1：表达式变换的多维标签预测

#### 目标

验证模型是否能识别一条变换具体改变了什么。

#### 数据

1. 用 SymPy 自动生成表达式族：
   - 多项式
   - 有理式
   - 三角式
   - 初等函数组合
2. 用规则引擎生成：
   - 等价变换对
   - 非等价变换对
   - 轨迹序列

#### 标签

- `truth_preserved`
- `domain_preserved`
- `operator_changed`
- `structure_changed`
- `explicitness_delta`
- `complexity_delta`

#### 基线

1. 纯规则基线
2. 纯序列模型基线
3. AST 图模型基线
4. 混合模型

#### 指标

- 多标签分类 F1
- truth-preservation accuracy
- explicitness 排序相关系数
- 错误类型统计

### 实验 2：欧氏 vs 双曲的 Mathlib 图谱嵌入

#### 目标

验证“形式化数学知识图谱具有双曲友好性”。

#### 基线模型

1. Node2Vec / DeepWalk
2. GCN / GraphSAGE
3. Poincaré Embedding
4. HGCN / Hyperbolic Graph Transformer

#### 指标

- MRR
- Hits@k
- MAP
- 低维下的结构失真
- 层级检索质量

#### 期望结果

如果在 16 维或 32 维时双曲模型显著优于 128 维欧氏模型，那么这将成为整篇项目最重要的论据之一。

### 实验 3：性质 probing

#### 目标

验证 embedding 空间中是否出现了可解释的“性质方向”。

#### 任务示例

- 预测一个节点是否是 definition / theorem / class / instance
- 预测一个对象是否带有某类代数结构
- 预测某 theorem 更接近“代数”还是“拓扑”局部区域

#### 指标

- linear probe accuracy / F1
- clustering purity
- 可视化可分性

### 实验 4：proof state 检索与 tactic 预测

#### 目标

验证图谱与 proof state 表示能否帮助证明搜索。

#### 基线

1. 纯文本 BM25
2. 欧氏向量检索
3. 双曲向量检索
4. 检索 + tactic ranking

#### 指标

- top-k tactic accuracy
- pass@k compile rate
- 检索命中率
- 证明长度变化

## 9. 推荐的模型策略

### 9.1 不建议一开始纯神经化

你的设想很强调“显式维度”，因此第一版模型不应完全交给黑箱网络。

更合理的做法是：

1. 用规则和静态分析先构造一批可靠特征。
2. 用神经网络去学习这些特征之间的耦合关系与低维表示。
3. 用解耦损失约束不同子空间不要完全纠缠。

### 9.2 一个可行的第一版混合模型

对于表达式级实验，可以采用：

1. **符号分支**
   - AST encoder
   - operator histogram
   - variable-role feature

2. **语义分支**
   - 数值采样一致性
   - 域约束检查
   - 简单 SMT/SymPy equivalence oracle

3. **解耦头**
   - truth head
   - structure head
   - operator head
   - complexity head
   - explicitness head

4. **损失函数**
   - 多标签监督损失
   - 对比学习损失
   - 子空间去相关损失

### 9.3 对图谱实验的建议

图谱实验中不要急着做自然语言对齐。

正确顺序应为：

1. 先只做 formal graph
2. 再做欧氏/双曲对比
3. 再做 property probe
4. 最后才尝试自然语言 query 对齐

否则会把“表示问题”和“语言对齐问题”搅在一起，难以定位失败原因。

## 10. 评价指标与成功标准

### 10.1 最小成功标准

如果满足以下条件，我会认为第一阶段实验是成功的：

1. 表达式变换审计器在合成数据上稳定地区分：
   - 真值保持
   - 条件保持
   - 运算变化
   - 显式度变化
2. Mathlib 图谱能成功抽取，并完成一个可复现的欧氏/双曲基线实验。
3. proof state 检索至少在一个小规模数据集上优于纯文本检索基线。

### 10.2 较强成功标准

如果还能做到下面这些，项目就已经具备很强的论文潜力：

1. 双曲模型在低维下稳定优于欧氏模型。
2. 嵌入空间中出现可解释的性质方向或局部簇。
3. proof-hint 检索能实际提升 tactic 成功率或 pass@k。

## 11. 阶段性执行方案

### 阶段 0：准备期，2 到 3 周

目标：跑通环境，明确数据接口。

任务：

1. 搭建 Python 环境
2. 安装 SymPy、PyTorch Geometric、Geoopt
3. 跑通 Lean 4 与 LeanDojo
4. 明确数据落盘格式

产出：

- 环境说明
- 数据接口说明
- 第一个可运行脚本

### 阶段 1：表达式级最小闭环，3 到 5 周

目标：先做最容易形成闭环的系统。

任务：

1. 定义表达式语言子集
2. 编写规则生成器
3. 生成正负样本
4. 实现规则基线和简单学习基线
5. 输出一份“变换审计报告”

产出：

- 合成数据集
- 第一版多维标签体系
- 实验 1 初步结果

### 阶段 2：形式化图谱抽取，4 到 6 周

目标：把 Lean 资源变成可训练对象。

任务：

1. 从 LeanDojo 提取 node-edge 数据
2. 规范节点类型和边类型
3. 统计图规模、层级深度、连通特征
4. 先跑欧氏基线

产出：

- Mathlib 图谱数据
- 欧氏基线实验
- 图统计报告

### 阶段 3：双曲嵌入与 probing，4 到 6 周

目标：验证你的核心几何假设。

任务：

1. 实现 Poincaré 或 HGCN 基线
2. 对比低维双曲与高维欧氏
3. 做性质 probing 和可视化

产出：

- 实验 2 和实验 3 结果
- 一组能支撑论文 story 的图表

### 阶段 4：proof state 与 tactic 实验，6 到 8 周

目标：把表示学习接到形式化证明流程上。

任务：

1. 抽取 proof state 与 tactic trace
2. 定义 tactic 类型标签
3. 构造近邻检索器
4. 比较纯文本检索与向量检索

产出：

- proof state 检索基线
- tactic top-k 结果
- Lean 验证闭环样例

### 阶段 5：集成 Demo，3 到 4 周

目标：形成可展示的原型。

输入一个表达式或 Lean goal，系统能够：

1. 展示多维特征
2. 显示候选近邻
3. 给出候选变换或 tactic
4. 用验证器判断是否合法

## 12. 建议的论文故事线

如果目标是一年内产出一个小成果，最可行的故事线不是“自动证明难猜想”，而是：

**我们提出一种面向形式化数学对象的显式多视角表示框架，并证明：**

1. 局部表达式变换可以被分解为多个可解释维度上的变化；
2. 形式化数学图谱在双曲空间中具有更好的低维表示性质；
3. 这种表示可用于 proof-hint retrieval，而不是只做静态可视化。

这条故事线比“直接证明欧拉常数无理性”更稳，也更容易形成实验闭环。

## 13. 当前阶段不建议做的事情

以下内容不应在第一阶段启动：

1. 从全网数学书自动形式化生成图谱
2. 端到端自动生成长证明
3. 直接把 `γ` 无理性或黎曼猜想当成主任务
4. 一开始就把自然语言检索、双曲检索、proof planning 全部堆在一起

原因很简单：这些方向都太容易让失败原因不可分。

## 14. 主要风险与应对

### 风险 1：真值标签不可靠

应对：

1. 限制表达式语言
2. 采用多重 oracle  
   包括规则、符号化简、数值采样、域检查

### 风险 2：LeanDojo 数据抽取成本高

应对：

1. 先抽小子集
2. 先做 graph statistics
3. 再决定是否扩展到全量 Mathlib

### 风险 3：双曲训练不稳定

应对：

1. 先复现 Poincaré embedding 基线
2. 控制维度和学习率
3. 先验证简单图，再上大图

### 风险 4：proof state 过于复杂

应对：

1. 先做检索，不做生成
2. 先做 tactic 类型预测，不做完整证明搜索

## 15. 结论

从这份对话中提炼出来的最重要结论是：

**这个方向要想真正工程化，关键不是先去追逐“自动证明难题”，而是先把“数学对象的多视角显式表示”做成一个可验证、可比较、可检索的基础设施。**

如果把项目拆成：

1. 表达式级多维变化分析
2. 形式化图谱的双曲表示
3. proof state 的轨迹检索

那么它就从一个抽象的数学哲学设想，变成了一个清晰的实验研究计划。

对当前阶段而言，最优先的起点有两个：
